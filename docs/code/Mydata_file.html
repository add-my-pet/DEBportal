The mydata file is where one sets the metadata, data, auxiliary data, text for data (units, label, bibkey, comment), weight coefficients and options for grouped plotting.

One can either expand upon an existing mydata file or else create a new mydata file from an empty template. In your local DEBtool folder go to DEBtool_M\lib\pet and you will find zipped folders containing empty templates for the 'std' and the 'abj' models (templates_std.zip and templates_abj.zip) respectively. You can extract the template for a mydata file (which is independent of the model type), rename it and then start filling it out. If you start from a copy of an existing related species, please first remove all bibkeys and insert the correct one, after updating the value of the corresponding data. This is to make sure that you replace all data. See the collection for examples; the mydata-code can be seen for each entry via dropdown CODE on the toolbar of the entry.

Focus on getting data and seeing what type of label it has (see Zero-variate data and Uni-variate data) to get an overview on the 'data completeness' and also learn the nomenclature. Knowing the nomenclature will give you speed in perusing the AmP species list page and seeing where similar data to yours is found. It is useful for having standardized and consistent names for the data in the mydata file. Please see this youtube movie on how to add uni-variate data to a mydata file.


The next subsections present the structure of the mydata file following the order of appearance within the file. However, the metaData fields with Discussion points, Facts, Acknowledgement, Links and References are flushed to the bottom of the mydata-file.

Metadata[edit]

In this section of the file we fill in the metadata of the entry that will be saved into the metaData structure. All ecoCode fields are described in more detail, and their code is given, on the dedicated Add-my-Pet AmPeco webpage. The fields to be filled are presented in the following table on the top of the mydata-file, while some additional metadata fields are specified at the bottom of the mydata-file.


metaData fields Description
phylum taxonomic phylum
class taxonomic class
order taxonomic order
family taxonomic family
species taxonomic species, using underscore to separate genus from species, e.g. Homo_sapiens
species_en the (most popular) common name of the species
ecoCode.climate Specify a climate for your species by choosing a climate code defined in the AmpEco climate section. The climate is the pattern of variation in temperature, humidity, atmospheric pressure, wind, and precipitation, averaged over long periods of time. (AmP follows the augmented Köppen-Geiger climate classification.)
ecoCode.ecozone specify an ecological zone for your species following the ecozone codes defined in the AmpEco ecozone section. The ecozone is a large geographic area of the Earth's surface within which organisms have been evolving in relative isolation over long periods of time. (AmP follows the augmented Udvardy ecozone classification.)
ecoCode.habitat Define a habitat for your species -- habitat codes are defined in the AmpEco habitat section. The habitat specifies the physical, chemical, and biological surroundings in which a particular species of organism lives. The habitat codes are meant to be combined with the other codes; the life stage code is prepended to the habitat code.
ecoCode.embryo Embryo environment (see AmpEco embryo environment code) specifies the local environment during the embryo stage, partly involving parental care.
ecoCode.migrate Specify if your species migrates or goes into torpor (see AmpEco migration/torpor code). Empty code means no migration or torpor. For migrating species, habitat and ecozone refer to breeding area.
ecoCode.food The life stage code is prepended to the AmpEco food code.
ecoCode.gender Use the AmpEco gender code to specify if your species is a hermaphrodite (both male and female) or gonochoric (either male or female, also called dioecy).
ecoCode.reprod Use the AmpEco reproduction code to specify if your species has a single generation (homogamy: female, male, hermaphrodite), or has alternate generations, e.g. between sexual or asexual, known as heterogamy.
T_typical typical body temperature of the organism
data_0 vector with codes for zero-variate data used
data_1 vector with codes for univariate data used
COMPLETE level of the completeness of all the data
author name(s) of the author(s)
date_subm date of submission
email email of the contact author
address institutional address of the contact author
author_mod_i name(s) of the author(s) (add only if the entry concerns a modification of an existing one)
date_mod_i date of modification (add only if the entry concerns a modification of an existing one)
email_mod_i email of the contact author (add only if the entry concerns a modification of an existing one)
address_mod_i institutional address of the contact author (add only if the entry concerns a modification of an existing one)


For curators only:

make sure to add and fill in these fields before submitting to the person responsible for publishing the entry to the web

metaData fields Description
curator name(s) of the curator(s)
email_cur email of the contact curator
date_acc date of acceptance

Discussion points[edit]

The user should add strings with "discussion points", that gives interesting information on the data or results. If some parameters for males differ from that of females, please indicate which ones. If particular data is ignored during estimation (by setting the corresponding weights to zero), please give arguments. The discussion points are labelled as D1, D2, etc.

These strings are packed in the metaData structure in the field discussion.

Facts[edit]

Any relevant additional facts from the literature can be added here.

Facts are labeled as F1, F2, etc. They are packed in the metaData structure in the field facts. A reference for each fact should be added to a bibkey field, like bibley.F1, bibley.F2, etc, in metaData.

Acknowledgement[edit]

In this optional section of the mydata-file, people or project numbers of supporting funding agencies can be acknowledged.

Links[edit]

The metaData.links field needs ID codes (sometimes a species name, or a number) for a set of general and taxon-specific websites. These ID's can be copied from the address-field of the html-page for that species. For example, on Catalogue of Life, for mydata file use only the sequence of numbers and letters after "/id/" (a0b8ed2779121cee709ec4b1b05b6a30 from http://www.catalogueoflife.org/col/details/species/id/a0b8ed2779121cee709ec4b1b05b6a30). Webpage-specific formats are listed below. Function get_links (without input) opens all AmP-supported websites. Find your species in the search-boxes on these sites.
General websites are 

  links.id_CoL   for Catalogue of Life (Search), (use sequence of numbers and letters after "/id/")  
  links.id_EoL   for Encyclopedia of Life, (use sequence of numbers after "/pages/")
  links.id_Wiki  for Wikipedia, (use scientific_name or common_name after "/wiki/") 
  links.id_AWD   for Anim. Diversity Web, (use scientific_name between "/accounts/" and "/classification/")
  links.id_Taxo  for Taxonomicon, (use sequence of numbers after "?id=")
  links.id_AnAge for AnAge.  (use scientific_name after "entry.php?species=") 
  links.id_WoRMS for WoRMS. (use sequence of numbers after "?id=")

Taxon-specific websites are 

  links.id_molluscabase for MolluscaBase,  (use sequence of numbers after "?id=")
  links.id_fishbase     for fishbase, (use scientific-name.html, i.e., everything after "summary/")
  links.id_amphweb      for Amphibiaweb,  (use scientific+name after "?where-scientific_name=")
  links.id_ReptileDB    for ReptileDB, (use genus=scientific&species=name  after "/species?")
  links.id_avibase      for Avibase,  (use sequence of numbers and letters after "?avibaseid=")
  links.id_birdlife     for Birdlife,(use common-name-scientific-name  after  "/factsheet/")
  links.id_MSW3         for Mammal Spec World, (use sequence of numbers after "?id=")

If a species is not on the website, you can link to the genus or the id can be omitted, and the link will not appear in the links-dropdown in the toolbar of the entry-pages. Otherwise, please be as complete as possible. Make sure that the name of your entry is accepted by CoL.

References[edit]

For every bibkey code for data or facts a reference should be listed here using the standard types and fields one can find in the Bibliographic information file. Use of doi (digital object identifiers) is strongly encouraged

The references are packed in the field biblist in metaData.

Data[edit]

Data is separated into two types zero-variate (i.e. a single number in the preferred sequence time, length, weight, rate) and univariate data (i.e. a set of pairs of numbers) and are set in that order. Please follows the nomenclature of data-types as close a possible (but Matlab does not accept "-" in a name: tL stands for a unvivariate data set with time and length in the columns, tL_f for that of females and tL_M for that of males. Observe, for instance, that a_p means "age at puberty", i.e. time since the start of development at puberty, while t_p means "time since birth at puberty". Wet weights are indicated with Ww, dry weights with Wd, etc. The preferred units are d, cm, and g.

Text for data[edit]

Each data set need to have a corresponding text field for units, label, bibkey and, optionally, a comment with e.g. a remark on the accuracy, or how the data is obtained. Examples of frequently used comment fields: "temperature was guessed" or "egg wet weight was computed from egg diameter" or "ref ... gives a different value" or "the reported range is ..." or "the length is taken from snout to vent". The latter might also be made explicit in the label, like "standard length, total length, SVL". This text is set close to the data that it corresponds to.

Text for data is packed to output in the section "Pack auxData and txtData".

Auxiliary data[edit]

Examples of auxiliary data sometimes needed are presented in the following table:

auxData fields Description
temp temperature
length initial lengths
JX ingestion fluxes
food food availabilty

All data that have time in its units, require an auxData field of the type "temp". Warning: make sure the field name of the auxData is not the same as a field name in structure par or cPar .

When auxiliary data is needed, it is set close to the data that it corresponds to. And in the same way as data, not only the value must be provided, but also the corresponding units and label.

Auxiliary data is packed to output in the section "Pack auxData and txtData".

Pack auxData and txtData[edit]

A technical section of the mydata-file that ensures that the data is properly packed for output.

Weight setting[edit]

Each data point has to be assigned a weight for the estimation procedure. The weights are set in the structure weights.

Usually an automatic assignment is used, but the user can overwrite the assigned values. The reasoning for overwriting the values should be solid and noted in a commented line.

Pseudo-data setting[edit]

Pseudo-data is added to the data structure, together with the corresponding units and label.

Plot options[edit]

This optional section of the mydata-file can be used to define that some of the data sets should be plotted in the same figure; the labels of plots in the same figure should match exactly. The colours of the data points and predictions follow the lava-colour scheme (from white to black, via red and blue). So if data concerns females and males, plot them in this sequence to ensure the females are plotted in red, males in blue and mention in the title "Data for females, males". If data concerns different temperatures, plot them from high to low temperatures and mention the temperatures on the title (see the collection for examples). If nothing is set here the graphs for each univariate data set will be automatically plotted separately. If a file with the name custom_results_group.m is present in the current directory, the default plotting is skipped and this file is used to present graphics. See custom_results_template.
